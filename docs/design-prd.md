# Lecture Study Sidebar 界面重设计需求文档

## 1. 项目背景

Lecture Study Sidebar 是一个 Chrome 扩展，运行在 YouTube 视频页上，帮助用户把长课程视频转换成可学习、可检索、可保存的学习资料。当前功能已经基本完成，但现有界面信息层级混乱、视觉粗糙、占用视频空间较多、加载状态和分析状态不够直观，因此需要重新设计一套更简洁、更专业、更适合长期学习使用的界面。

这份文档用于交给 AI 设计工具生成新 UI。设计工具不需要重新定义功能，也不要删减功能；重点是重新组织信息架构、界面布局、交互状态和视觉风格。

## 2. 产品定位

产品名称建议：

- 中文：课程学习侧边栏
- 英文：Lecture Study Sidebar

一句话定位：

在 YouTube 课程视频旁边自动生成结构化大纲、画面分析和课程问答，并把学习结果保存成本地资料库。

目标用户：

- 在 YouTube 上学习 MIT OpenCourseWare 或其他公开课的学生、自学者、研究人员。
- 需要快速理解长视频结构，而不是完整重看每一分钟内容的用户。
- 需要保存课程资料，后续检索、复习和整理知识库的用户。
- 中文和英文用户都要支持。

核心价值：

- 自动把长视频变成可读学习大纲。
- 分开处理「字幕讲解内容」和「PPT/板书/白板/屏幕画面内容」。
- 用户可以围绕视频内容提问，答案基于视频字幕、大纲、画面文字和公开参考资料。
- 用户生成的数据保存在本地目录，方便后续检索和备份。

## 3. 设计目标

本次设计的目标是把扩展做成一个干净、克制、专业的学习工具，而不是一个调试面板。

必须做到：

- 用户第一眼看到的是「已经生成的学习内容」，不是日志、原始字幕或工程状态。
- 侧边栏必须可以收起，避免长期遮挡视频。
- 侧边栏内部必须自己滚动，不能把 YouTube 页面一起滚动。
- 大纲生成、画面扫描、问答生成都要有明确进度和状态反馈。
- 中文和英文界面都要自然，不能因为英文长文案或中文密集文本导致布局崩坏。
- 正常用户不应该看到调试信息、原始 JSON、开发命令或内部实现细节。

不希望出现：

- 大面积按钮堆叠。
- 大量 debug log 常驻展示。
- 把字幕原文作为主界面内容。
- 需要用户反复点击「获取字幕」「生成学习包」这类多余步骤。
- 视觉上像默认表单、默认浏览器插件、默认后台管理系统。

## 4. 核心使用场景

### 4.1 首次使用

用户安装扩展后，点击插件图标进入轻量 popup。

Popup 需要提供：

- 打开 YouTube/MIT 课程搜索入口。
- 打开设置页入口。
- 简短说明：需要先配置 DeepSeek API Key，建议选择本地保存目录。

Popup 不承担主要学习功能，不应该做复杂设计。

### 4.2 配置设置

用户进入设置页后，需要配置：

- DeepSeek API Key。
- DeepSeek Base URL。
- DeepSeek 模型名称。
- 自动分析课程视频开关，默认关闭。
- 画面扫描间隔，默认 45 秒，可设置。
- 本地保存目录，通过系统 Finder/File Explorer 目录选择器授权。
- 界面语言：跟随浏览器、中文、English。
- 输出语言：跟随界面、中文、English。
- 笔记风格：讲义式笔记、结构化大纲、复习笔记。
- 侧边栏宽度。
- DeepSeek 连接测试。

设置页设计要求：

- 明确区分「必填配置」和「可选偏好」。
- API Key 缺失时要清楚提示，不要让用户到视频页才困惑。
- 本地目录不能手动输入路径，必须通过目录选择授权。
- 告诉用户正常使用不需要运行本地服务或 `npm run csv-server`。
- 解释自动分析默认关闭的原因：避免用户看娱乐视频或普通 YouTube 视频时自动消耗 API。

### 4.3 进入 YouTube 视频页

扩展在 YouTube watch 页面注入右侧学习侧边栏。

默认行为：

- 如果自动分析关闭，显示「已进入视频，自动分析已关闭，需要时点击生成大纲」。
- 如果自动分析打开，进入视频后自动读取课程内容、生成大纲、启动画面扫描。
- 如果当前不是视频页，显示轻量提示，不展示空的复杂界面。

侧边栏必须支持：

- 收起。
- 重新展开。
- 在普通 YouTube 页面工作。
- 在视频全屏时仍可使用。
- 不遮挡过多视频内容。
- 宽度遵循设置项。
- 内部内容独立滚动。

### 4.4 自动生成大纲

大纲基于 YouTube 字幕/文字记录读取，然后调用 DeepSeek 生成。

大纲生成流程：

- 读取 YouTube 页面中的课程文字记录。
- 如必要，尝试 YouTube transcript 接口。
- 如必要，尝试播放器字幕轨道兜底。
- 长视频会被拆成多个片段发给 DeepSeek。
- 每完成一个片段，就在界面中 progressive 渲染部分大纲。
- 最终结果会合并、去重、按时间排序。
- 输出语言必须跟随用户设置，例如用户设置中文时，大纲必须是中文。

界面要求：

- 进度条不能只显示百分比，还要显示当前阶段。
- 用户应该知道是正在「读取课程内容」「分析第几段」「整理保存」还是「已完成」。
- 已生成的部分要立刻展示，不能让用户长时间只看到 loading。
- 每个大纲小节要有时间戳和跳转按钮，点击后跳到视频对应时间。
- 大纲内容应是主视图中最重要的信息。

### 4.5 画面分析

画面分析和字幕大纲是两类内容，必须在 UI 上清楚区分。

当前实现逻辑：

- 不是像字幕一样一次性通过 API 获取全部画面。
- 插件会在视频播放过程中按间隔截取当前可见画面。
- 首次约几秒后开始扫描，之后按用户设置的间隔继续扫描。
- 如果检测到广告，暂停画面分析，避免把广告截图和广告关键词保存进去。
- 如果标签页切到后台，隐藏标签页无法可靠截图，需要显示状态，回到前台后补扫。
- 本地模型先判断当前帧是否是课程关键画面。
- 关键画面类型包括 PPT、板书、白板、屏幕共享、普通视觉画面。
- 浏览器本地 OCR 提取画面文字。
- DeepSeek 只接收 OCR 文本和附近字幕上下文，不接收图片。
- 分析结果保存到画面 tab，并写入本地文件。

画面分析界面要求：

- 用户需要看到这是「定时扫描」，不是一次性全量解析。
- 显示当前播放时间、上次扫描时间、下次预计扫描时间、扫描间隔。
- 明确显示广告暂停、后台暂停、正在扫描、跳过非关键画面、OCR 为空、分析完成等状态。
- 画面结果要按时间排序，每条结果包含时间戳、画面类型、关键词/标签、提取到的画面原文、AI 分析结论。
- 对 PPT、板书、白板等不同类型要有清晰标签，不要把所有东西都叫 PPT。
- 可以展示关键帧预览，但预览不应喧宾夺主。

### 4.6 视频问答

用户可以在问答 tab 中围绕当前视频提问。

问答依据：

- 当前视频字幕。
- 已生成大纲。
- 画面 OCR 和画面分析。
- 公开参考资料：通过 DuckDuckGo Instant Answer 获取的公开片段。

问答规则：

- 优先依据视频本身内容。
- 如果公开视频内容和外部资料冲突，以视频内容为准。
- 能引用时间戳时，要展示时间戳。
- 如果证据不足，要直接说明缺少什么信息。
- 回答语言跟随输出语言设置。

问答界面要求：

- 输入框要易用，适合输入自然语言问题。
- 空状态要提示用户可以问什么，例如「这一讲的核心结论是什么？」。
- 回答时要有清楚的 thinking/loading 状态。
- 回答结果要分成「回答」和「依据来源」。
- 来源可以包括字幕、大纲、画面分析和公开 URL。

### 4.7 本地资料库

插件会保存每个视频的学习结果，用户可以在资料库 tab 查看最近课程。

资料库功能：

- 展示最近保存的课程。
- 支持搜索课程标题、大纲内容。
- 支持打开原视频。
- 支持从资料库加载已保存内容。
- 支持清空最近记录。

保存位置：

- 用户在设置页选择一个本地保存目录。
- 生成完成后自动保存，不需要用户手动导出。
- 如果没有选择目录，仍可在浏览器缓存中展示，但要提示没有保存到正式本地目录。

本地保存格式：

- 根目录：`lecture_library.csv`。
- 每个视频目录：`lectures/<video_id>/lecture.json`。
- Markdown 学习资料：`lectures/<video_id>/study_pack.md`。
- 图片目录：`lectures/<video_id>/images/`。

CSV 字段包括：

- `video_id`
- `video_url`
- `video_title`
- `published_at`
- `watched_at`
- `watch_time_seconds`
- `duration_seconds`
- `saved_at`
- `transcript_line_count`
- `transcript_text`
- `transcript_json`
- `visual_analysis_json`
- `summary`
- `tags`
- `tags_json`
- `outline_json`
- `lecture_notes_json`
- `concepts_json`
- `questions_json`
- `study_pack_markdown`

图片保存方式：

- 画面预览和 OCR 区域预览以图片文件保存到 `images/` 目录。
- JSON/CSV 中保存相对路径，不长期保存 base64。
- 这样比把 base64 直接塞进 CSV/JSON 更适合检索、备份和控制文件体积。

## 5. 信息架构

主要界面是 YouTube 右侧侧边栏。

建议侧边栏分为：

- 顶部：视频标题、当前状态、收起按钮、设置入口。
- 主导航：大纲、画面、问答、资料库。
- 主内容区：根据 tab 展示内容，内部独立滚动。
- 底部可选区域：导出 Markdown、复制、保存状态，不要占用太多空间。

Tab 定义：

- 大纲：主视图，展示 AI 生成的课程结构。
- 画面：展示 PPT/板书/白板/屏幕画面分析。
- 问答：围绕当前视频提问。
- 资料库：最近课程和本地缓存入口。

不要把以下内容放在主导航里：

- 原始字幕。
- 诊断日志。
- HTTP 请求状态。
- 原始 JSON。

这些只能作为隐藏的开发模式或完全移除普通 UI。

## 6. 关键状态设计

设计稿必须覆盖以下状态，不能只设计理想完成态。

### 6.1 全局状态

- 未进入 YouTube 视频页。
- 已进入视频页，自动分析关闭。
- 已进入视频页，自动分析开启，准备开始。
- DeepSeek API Key 缺失。
- DeepSeek 请求失败或额度/网络错误。
- 本地保存目录未选择。
- 本地保存目录权限失效，需要重新授权。
- 已保存到本地目录。
- 从本地缓存恢复。

### 6.2 大纲状态

- 空状态：尚未生成。
- 读取课程内容。
- 字幕/文字记录读取失败。
- DeepSeek 正在分析第 N/M 段。
- 部分大纲已生成，继续处理后续片段。
- 某个片段模型输出异常，已跳过并继续。
- 正在整理最终大纲。
- 正在保存到本地目录。
- 完成。

### 6.3 画面状态

- 自动分析关闭，不扫描画面。
- 等待视频画面。
- 当前页面有广告，暂停扫描。
- 正在截取当前画面。
- 正在本地判断画面类型。
- 首次使用正在下载本地视觉/OCR 模型。
- 正在本地提取画面原文。
- 当前画面不是新的课程关键画面，已跳过。
- 检测到关键画面但 OCR 为空，稍后重试。
- 正在用 DeepSeek 分析 OCR 文本。
- 画面分析已更新。
- 标签页在后台，回到前台后补扫。
- 本地画面分析失败。

### 6.4 问答状态

- 空状态：提示用户可以提问。
- 课程内容尚未准备，提问时会先读取课程内容。
- 正在结合字幕、画面和参考资料回答。
- 已回答。
- 问答失败。

### 6.5 资料库状态

- 没有保存过课程。
- 有最近课程列表。
- 搜索中无匹配结果。
- 资料库记录缺失或版本过旧，需要重新生成。

## 7. 视觉和交互方向

整体风格：

- 专业学习工具。
- 简洁、稳定、低干扰。
- 信息密度适中，适合长时间开在视频旁边。
- 不要娱乐化，不要过度科技感，不要像调试后台。

视觉建议：

- 采用清晰的内容卡片层级，但不要卡片过多。
- 大纲是主内容，字体层级要明显。
- 时间戳、状态、标签使用轻量视觉样式。
- 进度条要细、明确、有阶段文案。
- 错误提示要直接给出修复动作，例如「去设置填写 API Key」。

侧边栏交互：

- 展开态：默认贴右侧，宽度可配置。
- 收起态：只保留一个小型浮动入口，点击可展开。
- 全屏态：侧边栏应进入播放器全屏容器，不被浏览器全屏遮掉。
- 内容区独立滚动。
- Tab 切换不能重置当前已生成内容。

生成过程交互：

- 自动分析开启时，不要求用户再点「获取字幕」。
- 如果自动分析关闭，只保留一个主要行动按钮：「生成大纲」。
- 生成大纲后自动保存，不需要额外点击保存。
- 画面分析是持续进行的，应显示扫描节奏，让用户理解为什么后续画面还没出现。

## 8. i18n 要求

产品支持中文和英文。

需要设计：

- 中文界面。
- 英文界面。
- 跟随浏览器语言。
- 输出语言独立于界面语言。

布局注意：

- 英文按钮和状态文案通常更长，不能挤压变形。
- 中文大纲可能信息密度更高，要保证行距和段落可读。
- Tab 名称在中文和英文都应该短。

推荐 tab 文案：

- 大纲 / Outline
- 画面 / Visuals
- 问答 / Q&A
- 资料库 / Library

## 9. 隐私和安全约束

必须在 UI 或设置页中以简洁方式表达：

- 视频帧图片不会发送给 DeepSeek。
- 画面截图只在本地用于关键帧判断和 OCR。
- DeepSeek 只接收 OCR 文本、字幕上下文和用户问题。
- 课程数据默认保存在用户本地。
- 用户必须手动授权本地保存目录。
- 首次使用本地视觉/OCR 模型时，浏览器可能会下载模型文件，之后由 Chrome 缓存。

不要让用户误解：

- 不要暗示插件会上传完整视频。
- 不要暗示本地资料库保存在云端。
- 不要要求用户运行本地服务才能正常保存。

## 10. 页面和组件清单

AI 设计工具需要输出以下界面：

- YouTube 侧边栏展开态。
- YouTube 侧边栏收起态。
- YouTube 全屏模式下的侧边栏。
- 大纲 tab：空状态、生成中、部分结果、完成、错误。
- 画面 tab：扫描说明、扫描中、广告暂停、后台暂停、完成结果、错误。
- 问答 tab：空状态、输入中、回答中、回答完成、错误。
- 资料库 tab：空列表、有列表、搜索无结果。
- 设置页：完整表单、API Key 缺失提示、目录选择状态、连接测试状态。
- Popup：打开 YouTube 和设置入口。

关键组件：

- 顶部视频信息区。
- 全局状态条。
- 分段进度条。
- Tab 导航。
- 大纲小节卡片。
- 时间戳跳转按钮。
- 画面分析结果卡片。
- 画面扫描状态组件。
- 问答输入框。
- 问答结果和来源列表。
- 本地保存状态提示。
- 资料库列表项。
- 设置表单项。
- 错误提示和修复动作按钮。

## 11. 可访问性要求

- 所有按钮、tab、输入框必须可键盘操作。
- Tab 需要明确选中态。
- 错误状态不能只靠颜色表达。
- 文本对比度要适合长时间阅读。
- 侧边栏收起/展开按钮需要清晰的可访问名称。
- loading 和进度变化要有文本说明。

## 12. 非目标

本次设计不需要改变以下内容：

- 不重新设计字幕获取算法。
- 不重新设计 DeepSeek prompt。
- 不新增云端账号系统。
- 不做移动端 App。
- 不把本地资料库改成云同步。
- 不展示完整原始字幕作为主功能。
- 不把调试日志暴露给普通用户。

## 13. 设计验收标准

设计方案通过的标准：

- 用户打开视频后，能在 3 秒内理解当前是否会自动分析。
- 用户不需要理解字幕接口、OCR 模型、JSON 修复等技术细节。
- 大纲生成中，用户能看到当前阶段和已生成内容。
- 画面分析中，用户能理解它是按播放进度定时扫描，不是一次性解析完整视频。
- 用户能随时收起侧边栏继续看视频，并能可靠重新展开。
- 用户能清楚知道内容是否已保存到本地目录。
- API Key 缺失、本地目录未授权、广告暂停、后台暂停等异常状态都有明确提示和下一步。
- 中文和英文界面都保持可读、稳定、不拥挤。
- 普通用户界面中不出现调试日志、原始 JSON 或开发命令。
